## 4.3 정리

이번 장에서는 엔터프라이즈 어플리케이션에서 사용할 수 있는, 바람직한 예외 처리 방법에 대해 알아보았다. 주요 내용은 다음과 같다.

- 함부로 예외를 잡고, 조치를 취해주지 않거나 의미 없는 `throws`를 남발하는 것은 위험하다.
- 예외는 복구하거나, 예외 처리 오브젝트로 의도적으로 전달하거나, 적절한 예외로 변환 해 주어야 한다.
	- 좀 더 의미 있는 예외로 변환하거나, 클라이언트가 복구하기 힘든 예외 상황은 런타임 예외로 포장할 수 있다.
- 어플리케이션 로직을 담기 위한 예외는 체크 예외로 만든다.
- JDBC의 `SQLException`은 대부분 복구할 수 없는 예외이므로, 런타임 예외로 포장하는 것이 좋다.
- `SQLException`의 에러 코드는 특정 DB 접근 기술에 종속적이기에 DB에 독립적인 예외로 전환될 필요가 있다.
	- 스프링은 이를 위해 `DataAccessException`이라는, DB 접근 기술에 독립적인 런타임 예외 클래스를 제공한다.
- DAO를 DB 접근 기술에서 독립 시키려면 인터페이스의 도입, 런타임 예외 전환과 기술에 독립적인 추상화된 예외 사용이 필요하다.
